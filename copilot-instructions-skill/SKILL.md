# Copilot Instructions 生成器

分析游戏项目源码，提取关键信息，自动生成完整的 `.github/copilot-instructions.md` 协作指南文件。

## 适用场景

- 新项目初始化时，需要生成 copilot-instructions.md
- 项目发生较大改动后，需要更新 copilot-instructions.md
- 用户明确要求生成或刷新协作指南

## 分析步骤

按以下顺序分析项目代码：

### 第一步：识别游戏概述

- 从组件名称、注释、README（如有）中提取游戏名称和描述
- 从交互逻辑中推断核心玩法
- 从内容/题目数据中推断教学目标

### 第二步：提取游戏规则

- 从条件判断逻辑中提取胜利/失败条件
- 从分数相关代码中提取计分规则
- 从事件处理函数中提取交互规则
- 从 if/else、guard clause 中提取边界规则
- 识别 3 条最核心的规则，标注为「关键规则」

### 第三步：分析组件架构

- 绘制组件树（父子嵌套关系）
- 识别状态管理方案（useState/useReducer/Context/其他）
- 追踪 props 和 state 的数据流向
- 列出所有核心文件及其职责

### 第四步：提取实体定义

- 从 TypeScript 类型/接口定义中提取实体属性
- 从状态变量中提取状态枚举
- 从事件处理中提取实体间交互关系

### 第五步：识别约束

- 从 package.json 提取外部依赖
- 从代码中识别性能相关处理（防抖、节流、memo 等）
- 从配置文件中提取平台约束

## 需要读取的文件

执行此 skill 时，**必须**读取以下文件（如果存在）：

1. `index.tsx` — 主入口/主组件（**必须**）
2. 其他 `.tsx` / `.ts` 组件文件（如有拆分组件）
3. TypeScript 类型定义文件（如有独立的 `types.ts`）
4. `package.json` — 依赖与构建脚本（**必须**）
5. `vite.config.ts` / `webpack.config.js` 等构建配置
6. `tsconfig.json` — TypeScript 配置
7. `README.md` — 项目说明（如有）
8. `index.module.css` — 样式文件（可浏览结构）

## Bug 修复协议

收到问题后，按以下流程执行：

### 第一步：定位

- 根据现象/报错，定位到具体组件和代码段
- 说明根本原因（root cause），不仅是表面现象
- 引用「游戏规则」中的相关条目，确认理解一致

### 第二步：方案

- 说明要修改哪些文件的哪些部分
- 列出本次修改可能影响的关联模块
- 多种方案时简要对比，推荐最优

### 第三步：实施

- 输出修改后的完整代码段，包含足够上下文定位粘贴位置
- 用注释标注改动原因：`// FIX: [原因]`
- 不改动与本次修复无关的代码

### 第四步：自检

每次修复后逐项确认：

- [ ] 是否解决了报告的问题？
- [ ] 是否违反「游戏规则」中的任何条目？
- [ ] 是否影响「组件架构」中的其他组件？
- [ ] 状态流转是否仍然正确？
- [ ] 边界情况是否已处理（空值、极端输入、快速重复操作）？
- [ ] TypeScript 类型是否完整，无 any 逃逸？

任何一项不通过，标注 ⚠️ 并说明风险。

- 中文说明 + 英文代码
- 代码块标注文件路径
- 每次修复末尾附变更摘要：`📝 变更：[文件] - [改了什么] - [为什么]`
- 问题描述不清时，先列出理解和假设，确认后再动手
- 不确定的地方说"我不确定"，不编造 API 或行为

## 红线规则

- ❌ 不删除现有功能代码，除非明确要求
- ❌ 不更换技术方案（如状态管理库），除非它是 bug 根因
- ❌ 不做"顺便优化"，严格限定修复范围
- ❌ 不引入新依赖，除非讨论确认
- ⚠️ 涉及计分、胜负判定、关卡流转的修改，必须额外对照「游戏规则 > 关键规则」

## 输出要求

- 所有 `{占位符}` 必须替换为从代码中提取的**实际内容**
- 无法从代码中确定的信息，标注 `[待补充：需要确认 XXX]`
- 「Bug 修复协议」「输出规范」「红线规则」三个章节**原样保留**，不做任何修改
- 组件树使用实际的组件名称
- 游戏规则尽量完整，宁多勿少
- 关键规则标注 3 条最核心的、修改代码时最容易违反的规则

## 注意事项

1. 这是一个**分析+生成**任务，不是代码修改任务
2. 生成前必须完整阅读 `index.tsx`（全部行），不能遗漏逻辑
3. 如果项目有多个组件文件，全部读取后再开始生成
4. 生成的文件应能被 GitHub Copilot 和 Claude 等 AI 工具直接消费
5. 如果 `.github/copilot-instructions.md` 已存在，覆盖写入新内容
