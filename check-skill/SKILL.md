# Check Skill — 项目规则验证器

根据 `.github/copilot-instructions.md` 中定义的游戏规则和设计约束，对项目源码进行逐条自动化验证。用 AI 阅读代码+推理来代替人工走查，最大限度减少手动测试。

## ⚠️ 修复约束

当检查项 FAIL 需要修复代码时，必须遵守以下规则：

1. **最小化修复** — 只修改 FAIL 对应的代码，不做无关重构
2. **保护定制标记** — 代码中带有 `@custom-*` 标记的区域（如 `@custom-dev-panel`、`@custom-characters`、`@custom-bgm`、`@custom-sounds`）表示开发者已手工定制，修复时**不得覆盖或重写**这些区域的结构，只能在其内部做精确的最小修改
3. **修复前确认** — 如果修复涉及超过 10 行的改动，先说明方案再动手

## 核心理念

1. **规则即测试用例** — 从设计文档中提取每一条可验证的规则，转化为检查项
2. **能验则验** — AI 能通过阅读代码判定的规则，直接给出 PASS/FAIL
3. **不能验则标注** — 需要运行时环境、真实浏览器、或人工主观判断的，明确说明原因
4. **失败即修** — 发现 FAIL 不是记一笔就完，而是立即尝试修复，修复后记录到事件日志

## 执行前提

执行此 skill 前，**必须**完整读取以下文件：

| 优先级 | 文件 | 用途 |
|-------|------|------|
| **必须** | `.github/copilot-instructions.md` | 游戏规则的唯一真相源 |
| **必须** | `index.tsx`（全部行） | 主逻辑，验证的主要对象 |
| **必须** | `index.module.css` | 样式验证（布局、动画） |
| 推荐 | `package.json` | 依赖版本、构建脚本 |
| 推荐 | `tsconfig.json` | TypeScript 配置 |
| 推荐 | `vite.config.ts` | 构建输出格式 |

> 不要跳读或只读部分行。`index.tsx` 必须从第 1 行读到最后一行。

## 验证检查表

按以下分类逐项检查。A-D 类为**动态检查项**，从 `copilot-instructions.md` 的游戏规则中提取；E-G 类为**通用检查项**，所有项目适用。

### A. 游戏数据正确性（AI 直接验证 — 从设计文档提取）

从 `copilot-instructions.md` 的「游戏规则」和「角色与实体」章节提取所有可量化的数据约束。

**提取方法：** 逐段阅读设计文档，找出所有包含具体数值、枚举值、配置项的规则，为每条创建检查项。

| ID | 检查项模式 | 验证方法 |
|----|-----------|---------|
| A1 | 角色配置完整（数量、状态、主题色） | 检查 `CHARACTERS` 对象的键/值是否与文档一致 |
| A2 | BGM URL 完整（main/progress/rest 或文档定义的类型） | 检查 `BGM_URLS` 对象 |
| A3 | 关卡数量和参数匹配文档 | 检查 `LEVELS` 数组长度和配置项 |
| A4~N | {从文档中提取的其他数据约束} | {对应的代码位置和验证逻辑} |

> 示例：如果文档写"拼图为 3×4=12 块"，则检查 `PUZZLE_GRID = { rows: 3, cols: 4 }`。
> 示例：如果文档写"鱼群速度范围 1-5"，则检查相关常量范围。

### B. 核心规则逻辑（AI 直接验证 — 最关键）

从 `copilot-instructions.md` 的「游戏规则 > 基本流程」和「关键规则」章节提取所有行为约束。

**提取方法：** 找出所有描述"必须/不能/应该/禁止"的规则，以及条件-结果型规则（"当X发生时，Y必须执行"）。

| ID | 检查项模式 | 验证方法 |
|----|-----------|---------|
| B1 | 核心交互逻辑符合规则（点击/拖拽/碰撞等） | 检查对应事件处理函数的 guard 条件和执行路径 |
| B2 | 分数/评价计算逻辑正确 | 检查得分公式或星级判定逻辑 |
| B3 | 错误操作有即时反馈（视觉+音效） | 检查错误分支的 UI 更新和音效调用 |
| B4 | 正确操作有即时反馈 | 检查正确分支的 UI 更新和音效调用 |
| B5~N | {从文档提取的其他行为约束} | {对应的代码验证逻辑} |

> 示例：如果文档写"捕捉金色鱼得分，误捕扣分"，则检查 catch 处理函数的 fish.type 判断和分数加减。

### C. 状态机流转（AI 直接验证 — 关键规则）

从 `copilot-instructions.md` 的「组件架构」和「游戏规则 > 基本流程」中提取状态流转图。

**提取方法：** 找出所有 `onStatusChange(N)` 调用要求和屏幕/阶段切换顺序。

| ID | 检查项模式 | 验证方法 |
|----|-----------|---------|
| C1 | 状态枚举/类型覆盖所有阶段 | 检查 `type Phase` 或 `type GameState` 是否完整 |
| C2 | 流转顺序符合设计（不跳跃、不回退到已完成状态） | 追踪所有 `setPhase/setGameState` 调用链 |
| C3 | `onStatusChange` 按文档要求的顺序输出，不跳跃不重复 | 追踪所有 `onStatusChange(N)` 调用点 |
| C4 | 最终状态后调用 `onExit()` | 检查完成状态的 `onExit` 调用 |
| C5 | 完成后不再有状态变更 | 确认最终状态下无 `setPhase/setGameState` 调用 |

### D. 计时系统（AI 直接验证 — 如果有计时）

如果游戏有计时/倒计时机制，则检查以下通用项。无计时功能的游戏跳过此节。

| ID | 检查项 | 验证方法 |
|----|-------|---------|
| D1 | 计时器在正确时机启动 | 检查 `setInterval/setTimeout` 启动条件 |
| D2 | 暂停时计时器冻结 | 检查条件 `!isPaused` |
| D3 | 恢复后继续（非重置） | 确认 `togglePause` 只翻转布尔，不修改时间值 |
| D4 | 时间到时有正确处理（结束/进入下一阶段） | 检查 timeRemaining=0 时的处理逻辑 |
| D5 | 组件卸载时清除 interval/timeout | 检查 `useEffect` return 中的 `clearInterval/clearTimeout` |

### E. 音频系统（AI 直接验证）

| ID | 检查项 | 验证方法 |
|----|-------|---------|
| E1 | BGM 循环播放 | 检查 `playBGM` 中 `audio.loop = true` |
| E2 | 角色说话时 BGM 音量压低至 30% | 检查 `speak` 中 `bgmRef.current.volume = 0.3` |
| E3 | 角色说完话后 BGM 音量恢复至 100% | 检查 `speak` 回调中 `bgmRef.current.volume = 1.0` |
| E4 | 不同 BGM 切换时旧音频先停止 | 检查 `playBGM` 开头 `bgmRef.current.pause()` |
| E5 | 暂停时切换到休息 BGM | 检查 `togglePause` 中暂停时 `playBGM('rest')` |
| E6 | `audio.play()` 有 `.catch()` 容错 | 检查 `playBGM` 中 `.play().catch(...)` |
| E7 | 组件卸载时清理音频资源 | 检查 `useEffect` return 中 `pause + null` |

### F. 类型安全与代码质量（AI 直接验证）

| ID | 检查项 | 验证方法 |
|----|-------|---------|
| F1 | 无 `any` 类型逃逸 | 全文搜索 `: any`、`as any`、`<any>` |
| F2 | Props 接口正确：`{ params, onStatusChange, onExit }` | 检查 `ComponentProps` 定义 |
| F3 | `default export` 组件 | 检查文件末尾 `export default Component` |
| F4 | 使用 CSS Modules 避免样式冲突 | 检查 import style 和 className 使用方式 |

### G. 构建与平台约束（AI 直接验证）

| ID | 检查项 | 验证方法 |
|----|-------|---------|
| G1 | Vite 输出 CJS 格式 | 检查 `vite.config.ts` 中 `formats: ['cjs']` |
| G2 | React 设为 external | 检查 `vite.config.ts` 中 `external: ['react', 'react-dom']` |
| G3 | TypeScript target ES6 | 检查 `tsconfig.json` 中 `target` |
| G4 | JSX 模式 react-jsx | 检查 `tsconfig.json` 中 `jsx: 'react-jsx'` |

### G2. 模板一致性校验（AI 直接验证）

当项目根目录存在从 skill 模板复制的文件时，检查是否与模板源保持一致。不一致说明有一方被单独修改但未同步。

| ID | 检查项 | 验证方法 |
|----|-------|----------|
| G2-1 | `DevPanel.tsx` 与 `.claude/skills/debug-skill/templates/DevPanel.tsx` 一致 | diff 两个文件，不一致则 ⚠️ WARN |
| G2-2 | `DevPanel.module.css` 与 `.claude/skills/debug-skill/templates/DevPanel.module.css` 一致 | diff 两个文件，不一致则 ⚠️ WARN |
| G2-3 | 其他模板文件一致性（如 LevelSelectPage） | 扫描 `.claude/skills/**/templates/` 下的文件，检查项目根目录对应文件是否存在且一致 |

> 判定规则：
> - 两个文件完全一致 → ✅ PASS
> - 项目文件存在但内容不同 → ⚠️ WARN（提示用户运行 `sync-templates` skill 同步）
> - 模板文件存在但项目文件不存在 → ✅ PASS（可能尚未初始化，不报错）

### H. 需人工验证项（AI 无法验证）

| ID | 检查项 | 无法自动验证的原因 | 建议人工验证方法 |
|----|-------|--------------------|-----------------|
| H1 | 角色图片 URL 可访问、显示正确 | 需要 HTTP 请求+图片渲染，AI 无法发起 | 浏览器打开 URL 检查 |
| H2 | BGM URL 可正常播放 | 需要音频解码环境 | 浏览器播放测试 |
| H3 | SVG 渐变色连线视觉效果正确 | 需要 SVG 渲染引擎 | 浏览器运行第二关查看 |
| H4 | CSS 动画和过渡时长符合预期 | 需要实际渲染+计时 | 浏览器运行各关卡观察 |
| H5 | 移动端适配（vmin 单位）表现 | 需要多设备测试 | 用 Chrome DevTools 模拟不同屏幕 |
| H6 | 浏览器自动播放策略下音频恢复 | 依赖浏览器策略差异 | 在 Safari/Chrome 上分别测试 |
| H7 | 快速连续点击的防抖与状态一致性 | 需要真实用户操作速率 | 手动快速操作各关卡 |
| H8 | 游戏交互的视觉反馈用户体验 | 主观判断 | 人工体验并评价 |

## 执行流程

### 第一步：读取所有必需文件

按「执行前提」表逐个读取文件内容，**全部完成后**再开始验证。`index.tsx` 必须读完所有行。

### 第二步：逐项验证

按 A → B → C → D → E → F → G → H 的顺序逐项检查。对于每一项：

1. **定位代码**：找到与该检查项相关的代码段
2. **判定结果**：
   - ✅ **PASS** — 代码逻辑完全符合规则
   - ❌ **FAIL** — 代码逻辑违反规则（附上具体代码位置和原因）
   - ⚠️ **WARN** — 代码基本符合但存在隐患（附上风险说明）
   - 👤 **MANUAL** — 需要人工验证（仅 H 类）
3. **记录证据**：引用具体代码行号或代码片段作为判定依据

### 第三步：修复发现的 BUG

如果某项验证结果为 ❌ FAIL：

1. **先修复再继续** — 不要跳过，立即按 Bug 修复协议修复
2. **修复后重新验证该项** — 确认已变为 PASS
3. **记录到 BUG 事件日志**（见下方模板）
4. **检查修复是否影响其他项** — 如有关联，重新验证关联项

修复时严格遵守红线规则：
- 最小化变更，只改必须改的
- 用 `// FIX: [原因]` 注释标注改动
- 不做"顺便优化"

### 第四步：输出验证报告

## 扩展规则

如果项目的 `.github/copilot-instructions.md` 被更新（新增了规则或修改了规则），此 skill 的检查表也应同步更新。具体做法：

1. 读取最新的 `copilot-instructions.md`
2. 对比当前检查表，识别新增/变更的规则
3. 为新规则创建检查项，分配 ID，归入合适的分类
4. 在验证报告中标注 `[NEW]` 标记

## 与其他 skill 的关系

| Skill | 关系 |
|-------|------|
| `copilot-instructions-skill` | check-skill 的数据源 — 规则文档由它生成 |
| `debug-skill` | check-skill 不验证 dev 面板逻辑（非生产代码） |
| `group-skill` | 可将 check-skill 加入某个分组（如 `pre-release` 组） |

## 注意事项

1. **这是一个分析+验证+修复任务**，不是纯文档生成任务
2. 验证过程中如需修复代码，严格遵守 `copilot-instructions.md` 中的「Bug 修复协议」
3. 不要为了让验证通过而放松规则标准——如果真有问题，就标 FAIL 并修
4. `index.tsx` 必须完整阅读，不能遗漏任何函数
5. 对照检查时，代码的行为要与规则的**语义**一致，不只是表面相似
6. 每次验证的完整报告应直接输出到对话中，不创建额外文件（除非用户要求）
